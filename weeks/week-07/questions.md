1. Зачем нужен .proto?
Это «единый источник правды». Мы не пишем классы на Python вручную, потому что .proto позволяет автоматически генерировать код для любого языка (Java, Go, C++). Если сервис на Python должен общаться с сервисом на Go, им достаточно иметь один .proto-файл, чтобы идеально понимать друг друга.
2. Что такое Unary RPC?
Это самый простой тип вызова: один запрос — один ответ. Клиент отправляет одно сообщение, сервер возвращает один ответ. Концептуально это копия классического REST-запроса (GET или POST), где на один клик ты получаешь один результат.
3. Почему gRPC эффективнее REST (JSON)?
* Бинарный формат: JSON — это текст, он весит много. Protobuf переводит данные в байты, которые весят в разы меньше.
    * Экономия на именах: В JSON мы каждый раз передаем ключи "author": "s19". В gRPC вместо слова "author" передается просто цифра 3 (тэг).
    * HTTP/2: В отличие от старого HTTP/1.1, gRPC умеет сжимать заголовки и держать одно соединение для множества запросов.
4. Зачем нужны номера полей (тэги = 1)?
Они нужны для обратной совместимости. Если ты добавишь новое поле = 4, старый код его просто проигнорирует, но не сломается. Сервер находит данные не по имени поля (которое может измениться), а по его номеру. Поэтому номера полей в .proto нельзя менять после того, как сервис запущен.
5. Что такое Stub и Servicer?
* Stub (Стаб): Это «заглушка» или клиентский прокси. Для программиста это выглядит как обычный локальный метод, но внутри стаб берет данные, превращает их в байты и отправляет по сети.
    * Servicer: Это базовый класс на сервере. Он содержит «пустые» методы. Твоя задача — унаследоваться от него и написать реальный код (логику) внутри этих методов.
6. Что выбрать: REST или gRPC?
* REST: Если API публичный, для браузеров, или если важна простота отладки (чтобы можно было глазами увидеть JSON).
    * gRPC: Для внутреннего общения между микросервисами, где важна скорость, минимальные задержки и строгие контракты, которые не дадут программистам ошибиться в типах данных.
