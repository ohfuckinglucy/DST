# Вопросы для самопроверки

## 1. Какие критерии вы использовали при разделении монолита на микросервисы (или при проектировании сервисов)?

**Ответ:** Система проектировалась с нуля как микросервисная, поэтому разделение монолита не требовалось. Критерии выделения сервисов:

- **Одна ответственность (SRP):** Comments Service отвечает только за комментарии, Users Service — только за пользователей.
- **Независимость данных:** каждый сервис имеет своё in-memory хранилище, нет общей БД.
- **Независимость развёртывания:** сервисы могут запускаться и обновляться по отдельности.
- **Границы по домену:** комментарии и пользователи — разные сущности с разным жизненным циклом.

## 2. Почему вы выбрали именно этот стек технологий (язык, БД, протоколы)?

**Ответ:**

- **Python 3.11** — знакомый язык, богатая экосистема, асинхронность (asyncio).
- **FastAPI** — современный фреймворк с высокой производительностью (на уровне Node.js/Golang), встроенная Pydantic-валидация и автогенерация OpenAPI-документации.
- **REST (JSON)** — универсальный протокол, поддерживается любым HTTP-клиентом (curl, Postman, браузер). Для данного проекта (2 сервиса без интенсивного межсервисного общения) gRPC избыточен.
- **Nginx** — лёгкий, проверенный reverse-proxy, стандарт индустрии.
- **Docker + Docker Compose** — контейнеризация для воспроизводимости запуска одной командой.
- **In-memory хранилище** — достаточно для демонстрации; при необходимости заменяется на PostgreSQL/Redis.

## 3. Как ваши сервисы справляются с ошибками (retry, circuit breaker, graceful degradation)?

**Ответ:**

- **Валидация на входе:** Pydantic модели проверяют типы и структуру данных, при ошибке — HTTP 422.
- **404 Not Found:** при попытке получить/изменить/удалить несуществующий ресурс.
- **Healthchecks:** docker-compose проверяет `/health` каждого сервиса перед запуском gateway (через `depends_on` с `condition: service_healthy`). При падении сервиса Docker перезапускает контейнер (политика restart по умолчанию).
- **Graceful degradation** не реализована, т.к. сервисы независимы — отказ одного не влияет на другой.
- **Circuit breaker и retry** не используются, т.к. межсервисных вызовов нет (нет gRPC). При добавлении gRPC можно внедрить tenacity (retry) + circuitbreaker.

## 4. Как организован деплой и обновление вашей системы без простоя (Zero Downtime)?

**Ответ:** В текущей реализации (Docker Compose) zero-downtime не обеспечивается:

- При `docker compose up --build` старые контейнеры останавливаются, новые запускаются — есть downtime.
- Для zero-downtime требуется использовать Kubernetes (Rolling Update) или Docker Swarm. В проекте есть манифесты K8s (`k8s/deployment.yaml`), где настроен RollingUpdate с `maxSurge=1` и `maxUnavailable=0`, что позволяет обновляться без простоя.
- CI/CD пайплайн (GitHub Actions) автоматизирует сборку образов, но деплой на продакшен не настроен.

## 5. Что было самым сложным в интеграции всех компонентов вместе?

**Ответ:**

- **Согласование путей Nginx и сервисов:** пришлось добавить `rewrite ^/api/comments(.*)$ /comments$1 break;`, чтобы Nginx корректно проксировал запросы, убирая префикс `/api/comments` → `/comments` для FastAPI.
- **Healthcheck через Docker network:** сервисы обращаются к localhost внутри контейнера, а gateway — по имени сервиса. Пришлось настроить разные проверки.
- **Multi-stage Dockerfile:** подбор лёгкого базового образа (python:3.11-slim) и установка curl для healthcheck без увеличения размера.
- **Оркестрация зависимостей:** gateway должен ждать, пока оба сервиса станут healthy — это решено через `depends_on.condition: service_healthy`.

## 6. Что бы вы улучшили, если бы у вас был еще месяц?

**Ответ:**

1. **Добавить базу данных** — PostgreSQL вместо in-memory хранилища, с миграциями (Alembic).
2. **Добавить gRPC** для межсервисного общения (например, Comments проверяет существование пользователя через Users по gRPC).
3. **Реализовать аутентификацию и авторизацию** (JWT/OAuth2) — через middleware в Nginx или FastAPI.
4. **Настроить мониторинг** — Prometheus + Grafana для метрик, ELK для логов.
5. **Zero-downtime деплой** — настроить K8s-кластер (minikube или облачный) и CI/CD с автоматическим деплоем.
6. **Rate limiting и WAF** на Nginx.
7. **Написать нагрузочные тесты** (wrk/locust) и оптимизировать узкие места.
