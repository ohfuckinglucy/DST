Вот краткие ответы на вопросы для самопроверки по второй неделе:

### 1. Чем PUT принципиально отличается от PATCH? Какой из них является идемпотентным?
*   **PUT (Полное обновление):** Заменяет объект **целиком**. Клиент должен прислать все поля объекта. Если какое-то поле не передано, оно может быть удалено или обнулено на сервере (зависит от реализации).
    *   **Идемпотентность:** Да, идемпотентен. Отправка одного и того же PUT-запроса много раз приведет к одному и тому же результату (объект будет в одном и том же состоянии).
*   **PATCH (Частичное обновление):** Обновляет только те поля, которые были переданы в запросе. Остальные поля остаются без изменений. Удобно, например, для смены только статуса или имени.
    *   **Идемпотентность:** Не всегда. Зависит от логики. Если PATCH просто заменяет поле «название» на «Новое», то он идемпотентен. Но если PATCH делает операцию «увеличить счетчик на 1», то повторный вызов изменит результат, и он не будет идемпотентным.

### 2. Какие HTTP коды ошибок вы знаете и в каких ситуациях их следует возвращать?

#### **400 vs 422**
*   **400 Bad Request:** Общая ошибка. Сервер не может обработать запрос из-за синтаксической ошибки, неверного формата JSON или отсутствия обязательных заголовков. Клиент «сломал» протокол общения.
*   **422 Unprocessable Entity:** Специфическая ошибка валидации данных. Запрос syntactically правильный (JSON валиден), но **семантически** ошибочен. Например: поле `age` должно быть числом, а пришла строка `"двадцать"`; или поле `email` не содержит `@`. Это ответ Pydantic/FastAPI при неудачной валидации модели.

#### **401 vs 403**
*   **401 Unauthorized:** **«Кто ты?»**. Пользователь не авторизован (не logged in). Нет токена, истекла сессия или неверный пароль. Решение: представься (логин/пароль/токен).
*   **403 Forbidden:** **«Тебе нельзя»**. Пользователь авторизован (сервер знает, кто он), но у него **нет прав** на выполнение этого действия. Например, обычный пользователь пытается удалить админскую запись. Решение: никаких, права не дадут.

#### **404 Not Found**
*   Ресурс с указанным ID не существует в базе данных. Также может возвращаться, если сам URL-маршрут не найден (хотя чаще это 405 Method Not Allowed или 404 для пути).

### 3. Зачем нужны негативные тесты (тесты на ошибки)?
*   **Проверка устойчивости:** Гарантируют, что система не «падает» (не выдает 500 Internal Server Error) при некорректных данных от пользователя.
*   **Безопасность:** Убеждаются, что злоумышленник не сможет обойти проверки (например, получить доступ к чужим данным, подделав ID).
*   **Ясность контракта:** Проверяют, что сервер возвращает именно те коды ошибок (400, 404, 422) и сообщения, которые ожидает клиент для корректной обработки ситуации (например, чтобы показать пользователю понятное сообщение «Неверный формат email», а не «Произошла ошибка»).

### 4. Почему важно возвращать ошибки в структурированном формате (JSON), а не просто текстом?
*   **Автоматизация:** Клиентскому приложению (frontend, мобильное приложение) проще распарсить JSON `{"detail": "User not found", "code": "USER_404"}`, чем анализировать человеческий текст «Ошибка: пользователь не найден, попробуйте позже».
*   **Локализация:** Текст ошибки можно перевести на языке клиента на стороне фронтенда, используя код ошибки из JSON.
*   **Единый стандарт:** Все микросервисы в системе возвращают ошибки в одинаковом формате, что упрощает разработку общих обработчиков ошибок на клиенте.
*   **Детализация:** В JSON можно передать дополнительные поля (например, какое именно поле неверно: `{"field": "email", "error": "invalid format"}`), что невозможно сделать в простом тексте без сложного парсинга.

### 5. Что такое "обратная совместимость" API и как валидация помогает (или мешает) её поддерживать?
*   **Обратная совместимость:** Это свойство API, при котором новые версии сервера продолжают корректно работать со старыми клиентами, которые были написаны под предыдущие версии API. Клиенты не должны ломаться после обновления сервера.
*   **Как валидация помогает:** Она гарантирует, что данные строго соответствуют контракту. Если контракт не меняется, старые клиенты будут продолжать проходить валидацию.
*   **Как валидация может мешать:** Если ты ужесточишь валидацию (например, раньше поле `phone` принимало любые строки, а теперь только цифры в формате `+7...`), то старые клиенты, отправляющие данные в старом формате, получат ошибку 422 и сломаются.
    *   *Решение:* Чтобы сохранить совместимость, новые строгие правила часто вводят постепенно или делают поля опциональными, либо версионируют API (`/api/v1/...`, `/api/v2/...`).
