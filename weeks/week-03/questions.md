### 1. Forward Proxy vs Reverse Proxy

| Тип | Для кого работает | Что скрывает | Пример |
|-----|-----------------|--------------|--------|
| **Forward Proxy** | Для **клиентов** | IP-адреса клиентов от сервера | Корпоративный прокси, Tor, VPN |
| **Reverse Proxy** | Для **серверов** | Реальную структуру бэкенда от клиента | Nginx, API Gateway, Cloudflare |

**Простыми словами:**
- Forward Proxy: *"Я — клиент, скрой меня от интернета"*
- Reverse Proxy: *"Я — сервер, скрой мою инфраструктуру от клиентов"*

### 2. Что такое Upstream в Nginx?

**Upstream** — это логическое имя для группы бэкенд-серверов, на которые Nginx может распределять запросы.

```nginx
upstream backend_pool {
    server 10.0.0.1:8000;
    server 10.0.0.2:8000;
    server 10.0.0.3:8000 backup;  # резервный
}

server {
    location / {
        proxy_pass http://backend_pool;  # ← ссылка на upstream
    }
}
```

**Зачем:** балансировка нагрузки, отказоустойчивость, абстракция от реальных IP.

---

### 3. Зачем скрывать реальные IP-адреса сервисов?

- **Безопасность**: злоумышленник не увидит реальную топологию и не атакует сервис напрямую
- **Гибкость**: можно менять/масштабировать бэкенды без изменения у клиентов
- **Единая точка входа**: весь трафик идёт через Gateway, где можно применить политики (rate limiting, auth, логирование)
- **Сокрытие внутренней сети**: сервисы могут быть в приватной подсети без доступа извне

### 4. Задачи API Gateway кроме маршрутизации (минимум 2)

1. **Аутентификация/авторизация** — проверка JWT, API-ключей в одном месте
2. **Rate Limiting** — ограничение запросов от одного клиента для защиты от DDoS
3. **Кэширование** — отдача частых запросов без обращения к бэкенду
4. **Трансформация запросов** — изменение формата API, версионирование
5. **Логирование и мониторинг** — сбор метрик всего трафика в одной точке
6. **Терминация SSL/TLS** — расшифровка HTTPS на шлюзе, разгрузка бэкендов

### 5. Что если Gateway упадет? Как защититься?

**Последствия:** Все клиенты потеряют доступ ко всем сервисам (единая точка отказа — SPOF).

**Способы защиты:**

| Метод | Как работает |
|-------|-------------|
| **Репликация** | Запустить 2+ экземпляра Gateway + балансировщик перед ними (Keepalived, HAProxy) |
| **Health Checks** | Nginx/K8s автоматически исключают упавшие ноды из ротации |
| **Circuit Breaker** | При сбоях бэкенда — быстро возвращать ошибку, не "вешая" запросы |
| **Graceful Degradation** | Возвращать кэшированные ответы или заглушки, если бэкенд недоступен |
| **Kubernetes Ingress + ReplicaSet** | Авто-перезапуск упавших подов, горизонтальное масштабирование |

> Gateway должен быть **stateless** (не хранить состояние), чтобы любой экземпляр мог обработать любой запрос.
