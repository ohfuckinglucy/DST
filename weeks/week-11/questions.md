## Вопросы и ответы

### 1. Зачем создавать пользовательскую сеть (bridge network), если есть дефолтная?

| Причина | Объяснение |
|---------|-----------|
| **DNS-разрешение по имени** | В кастомной сети сервисы видят друг друга по имени: `ping db`, `http://app:8221`. В дефолтной — только по IP. |
| **Изоляция** | Сервисы в `app-net` не видят контейнеры из других проектов. Меньше шансов «случайно» подключиться не туда. |
| **Безопасность** | Можно явно указать, какие сервисы общаются между собой. |
| **Предсказуемость** | Дефолтная сеть может быть «замусорена» контейнерами из других проектов. |

```yaml
networks:
  app-net:
    driver: bridge  # создаём изолированную сеть
```

### 2. В чём разница между `build` и `image`?

| Параметр | Что делает | Когда использовать |
|----------|-----------|-------------------|
| `build: .` | Собирает образ из `Dockerfile` в текущей папке | Разработка: меняешь код → `docker-compose up --build` → готово |
| `image: nginx:alpine` | Берёт готовый образ из реестра (Docker Hub и др.) | Продакшен: образ уже собран, протестирован, версионирован |

```yaml
# Для разработки:
app:
  build: .

# Для прода:
app:
  image: myregistry/items-svc-s19:v1.2.3
```

### 3. Что делает `depends_on`? Гарантирует ли он инициализацию приложения?

❌ **Нет, не гарантирует.**

- `depends_on: [db]` означает лишь: *«запусти контейнер `db` перед `app`»*.
- Но контейнер может быть `started`, а PostgreSQL внутри — ещё не готов принимать соединения.

**Правильный подход** — ждать реальной готовности через `healthcheck`:

```yaml
depends_on:
  db:
    condition: service_healthy  # ждём, пока БД не скажет "я готова"
```

---

### 4. Как работают healthcheck и как использовать их в `depends_on`?

**Healthcheck** — это команда, которую Docker периодически выполняет, чтобы проверить, жив ли сервис.

```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8221/health"]  # команда проверки
  interval: 10s     # как часто проверять
  timeout: 3s       # таймаут одной проверки
  retries: 3        # неудачных попыток до статуса "unhealthy"
  start_period: 15s # время на "прогрев" при старте
```

**Использование в `depends_on`**:
```yaml
depends_on:
  db:
    condition: service_healthy  # app не запустится, пока db не станет healthy
```

> Статусы healthcheck: `starting` → `healthy` / `unhealthy`

### 5. Как передать переменные окружения в контейнер (минимум 2 способа)?

#### Способ 1: Прямо в `docker-compose.yml`
```yaml
environment:
  - DB_HOST=db
  - DB_PORT=5432
  - DEBUG=true
```

#### Способ 2: Через `.env`-файл (автоматически подгружается)
```bash
DB_PASSWORD=supersecret
```
```yaml
environment:
  DB_PASSWORD: ${DB_PASSWORD}
```

### 6. Почему Docker Compose редко используют в продакшене?

| Причина | Пояснение |
|---------|-----------|
| **Нет оркестрации** | Нет автоскейлинга, самовосстановления, rolling updates |
| **Единая точка отказа** | Всё на одной машине: упал хост — упало всё |
| **Слабый мониторинг** | Нет встроенных метрик, централизованного логирования |
| **Безопасность** | Сложнее реализовать RBAC, network policies, как в K8s |
| **Масштабирование** | `docker-compose up --scale` есть, но это «костыль» для прода |

**Но Compose идеален для**:
- Локальной разработки (`git clone` → `docker-compose up`)
- Тестовых сред и CI/CD
- Маленьких проектов на одном сервере (с должным харденингом)

---

```bash
# Сборка и запуск всего стека
docker-compose up --build

# Запуск в фоновом режиме
docker-compose up -d --build

# Просмотр логов
docker-compose logs -f app
docker-compose logs -f gateway

# Проверка здоровья сервисов
docker-compose ps

# Тестирование эндпоинта
curl http://localhost/api/items
```
