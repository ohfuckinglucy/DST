Вот краткие ответы на вопросы для самопроверки в формате Markdown:

### 1. В чем разница между "ресурсом" и "коллекцией" в контексте REST API?
*   **Ресурс** — это отдельный объект данных (например, конкретная задача с `id=5`). В URL он обычно представлен как `/tasks/5`.
*   **Коллекция** — это набор однородных ресурсов (список всех задач). В URL она представлена как родительский путь `/tasks`.
*   *Простыми словами:* Коллекция — это полка с книгами, ресурс — одна конкретная книга на этой полке.

### 2. Почему для создания объекта используется метод `POST` и код ответа `201`, а не `200`?
*   **POST** используется, потому что мы отправляем данные для обработки и создания нового состояния на сервере (не просто запрашиваем информацию).
*   **201 Created** — это специфический статус, означающий «запрос успешен, и в результате был создан новый ресурс».
*   **200 OK** означает просто «всё хорошо», но не гарантирует создание чего-то нового. Использование 201 дает клиенту четкий сигнал, что объект действительно появился в базе.

### 3. Что такое идемпотентность? Назовите HTTP-методы, которые являются идемпотентными.
*   **Идемпотентность** — это свойство операции, при котором многократное выполнение одного и того же запроса приводит к одинаковому результату (состояние сервера не меняется после первого успешного выполнения).
*   **Идемпотентные методы:** `GET`, `PUT`, `DELETE`, `HEAD`, `OPTIONS`.
*   **Не идемпотентный метод:** `POST` (если отправить запрос на создание дважды, появится два разных объекта).

### 4. Почему ID объекта обычно генерируется на сервере и возвращается в ответе на `POST`?
*   **Уникальность:** Только сервер может гарантировать, что ID не будет дублироваться в многопользовательской среде (избежание конфликтов).
*   **Безопасность:** Клиент не должен иметь права назначать свои идентификаторы, чтобы не нарушить целостность данных.
*   **Информирование:** Клиенту нужно знать присвоенный ID, чтобы в будущем обращаться к этому объекту (например, делать `GET /tasks/{id}`).

### 5. Что такое "контракт API" и почему важно его соблюдать?
*   **Контракт API** — это строгое соглашение о том, какие URL, методы, форматы данных (JSON) и коды ответов использует сервис.
*   **Важность:** Если сервер изменит формат ответа или название поля без предупреждения, все клиентские приложения (сайты, мобильные приложения), которые полагаются на этот контракт, сломаются. Соблюдение контракта обеспечивает стабильность взаимодействия между разными частями системы.

### 6. Зачем команде разработчиков нужна спецификация, например, OpenAPI (Swagger)?
*   **Единый источник истины:** Все разработчики (и фронтенд, и бэкенд) видят актуальное описание API в одном месте.
*   **Автоматическая документация:** Swagger UI генерирует интерактивную документацию автоматически из кода, где можно сразу тестировать запросы.
*   **Генерация кода:** На основе спецификации можно автоматически создавать клиентские библиотеки для разных языков программирования, экономя время на рутинную работу.
