## 3. Ответы на вопросы для самопроверки

| № | Вопрос | Ответ |
|---|--------|-------|
| 1 | В чём главное концептуальное отличие GraphQL от REST? | REST использует фиксированные URL-эндпоинты и возвращает заранее определённые структуры данных. GraphQL использует один эндпоинт, а клиент сам формирует запрос, указывая только нужные поля. Сервер возвращает ровно запрошенное, без лишних данных. |
| 2 | Что такое Schema Definition Language (SDL)? | Это декларативный язык описания схемы GraphQL. Он строго типизирован, определяет доступные типы, поля, аргументы, связи и обязательность значений. SDL выступает контрактом между клиентом и сервером. |
| 3 | Чем Query отличается от Mutation? Могут ли они выполняться параллельно? | `Query` предназначена для чтения данных и по спецификации должна быть идемпотентной. `Mutation` изменяет состояние сервера. В GraphQL запросы внутри `query` могут выполняться параллельно, а `mutation` выполняются последовательно, чтобы гарантировать порядок изменений. |
| 4 | Что такое "резолвер" (resolver) и кто его вызывает? | Резолвер — это функция, которая возвращает данные для конкретного поля схемы. Его вызывает GraphQL-движок (серверная библиотека) автоматически во время выполнения запроса, обходя дерево запроса. |
| 5 | Как решается проблема N+1 запросов в GraphQL (dataloader)? | С помощью паттерна **DataLoader**: он собирает запросы к одному источнику данных, группирует их в батч и выполняет один оптимизированный запрос вместо N отдельных. Также DataLoader кэширует результаты внутри одного запроса. |
| 6 | Почему в GraphQL сложно реализовать кэширование на уровне HTTP? | В REST каждый ресурс имеет уникальный URL, поэтому браузер/CDN кэшируют по URL + методам. В GraphQL все запросы идут на один URL (`/graphql`) с разным телом, поэтому стандартное HTTP-кэширование не работает. Используют: кэширование на уровне приложения, персистентные запросы (hashed queries), CDN с поддержкой GraphQL или клиентские кэши (Apollo, Urql). |
