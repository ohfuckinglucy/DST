1. Зачем нужны variables?
Чтобы отделить структуру запроса от данных. Это защищает от инъекций (злонамеренного кода), позволяет серверу кэшировать скомпилированный запрос и делает код чище. Склеивание строк через f-строки небезопасно и медленно.
2. Из каких полей состоит JSON-запрос?
Стандартно из двух:
* "query": строка с самим запросом (query или mutation).
    * "variables": словарь со значениями переменных.
    * (Иногда еще "operationName", если в строке несколько операций).
3. Почему только один URL?
Потому что GraphQL работает не с ресурсами (как адреса папок в REST), а с графом данных. Вы отправляете запрос на один эндпоинт (обычно /graphql), а сервер сам разбирает его содержимое и вызывает нужные функции (резолверы).
4. Что означает поле errors?
Оно содержит описание проблем, возникших при выполнении запроса (ошибки валидации, падение базы и т.д.). Да, оно может прийти вместе с data — это называется «частичный успех» (например, данные одного профиля нашлись, а другого — нет).
5. Как обрабатывать ошибки при статусе 200?
Нельзя доверять HTTP-статусу. Клиент должен обязательно проверять наличие ключа "errors" внутри тела JSON-ответа. Если он есть, нужно программно обработать ошибку, даже если сервер ответил «ОК».
6. Плюсы типизированных клиентов (Apollo/Relay):
* Автоматическое кэширование: они запоминают объекты по ID и не делают лишних запросов.
    * Синхронизация UI: если ты изменил имя юзера в одном месте, оно само обновится во всех компонентах экрана.
    * Генерация типов: они позволяют автоматически создавать интерфейсы (например, для TypeScript) на основе схемы.
